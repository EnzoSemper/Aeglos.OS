# Security Policy

## Supported Versions

| Version | Supported |
|---------|-----------|
| 0.1.x (current) | ✅ |
| < 0.1 | ❌ |

## Reporting a Vulnerability

**Do not report security vulnerabilities through public GitHub issues.**

Report security vulnerabilities by email to:

**security@aeglos.systems**

Include as much of the following as possible:

- Type of issue (e.g. capability bypass, memory corruption, inference poisoning, kernel escape)
- Component affected: kernel, Numenor, Ithildin, Aska, network stack, WASM sandbox
- Full path of the affected source file(s)
- Location of the vulnerable code (tag, branch, commit, or direct URL)
- Any special configuration required to reproduce the issue
- Step-by-step instructions to reproduce
- Proof-of-concept or exploit code (if available)
- Impact assessment — what an attacker could achieve

The more detail you provide, the faster we can triage and remediate.

## Response Timeline

| Stage | Target |
|-------|--------|
| Acknowledgement | Within 48 hours |
| Triage and severity assessment | Within 5 business days |
| Remediation and patch | Dependent on severity (see below) |
| Public disclosure | Coordinated with reporter |

### Severity-based remediation targets

| Severity | Target patch timeline |
|----------|-----------------------|
| Critical (CVSS 9.0+) | 7 days |
| High (CVSS 7.0–8.9) | 14 days |
| Medium (CVSS 4.0–6.9) | 30 days |
| Low (CVSS < 4.0) | 90 days |

## Scope

### In scope

- **Kernel capability system** — capability bypass, inheritance violations, privilege escalation
- **Syscall filter** — bypass of the per-task syscall allow-list
- **WASM sandbox** — escaping the sandboxed execution environment
- **Memory isolation** — cross-process VA access, TTBR0/TTBR1 boundary violations
- **Network stack** — TLS downgrade, certificate verification bypass, buffer overflows in TCP/IP
- **Numenor AI runtime** — prompt injection via tool calling, model-level attacks with kernel-level impact
- **Ithildin semantic memory** — unauthorised read/write of another process's stored content
- **ASLR bypass** — defeating the CSPRNG-seeded randomisation

### Out of scope

- Attacks requiring physical access to the host machine
- Vulnerabilities in upstream llama.cpp (report directly to the llama.cpp project)
- Issues in QEMU or the host OS
- Reports generated by automated scanners without proof of exploitability

## Security Design Notes

Reviewers should be aware of the following intentional design decisions:

- **No root user.** The capability model (`CAP_*` bitmasks) replaces Unix UID/GID entirely. There is no `CAP_ALL` path reachable from EL0.
- **TLS certificate verification is leaf-only.** Full CA chain verification is not yet implemented. Self-signed certificates will pass if the hostname and validity window match. This is a known limitation tracked for a future release.
- **Ollama / local LLM runs at EL0 with `CAP_AI`.** The inference engine is not in the kernel; it is a privileged userspace service. A compromised Numenor process cannot directly access kernel memory — it can only invoke AI syscalls it holds the capability for.
- **WASM sandbox has no JIT.** All third-party code is interpreted, not compiled to native — eliminating JIT-related speculative execution vulnerabilities.
- **ASLR applies to PIE (ET_DYN) binaries only.** Static ELF binaries (ET_EXEC) load at their link-time addresses. Use PIE compilation for security-sensitive EL0 components.

## Disclosure Policy

We follow coordinated disclosure. Once a fix is available we will:

1. Release a patched version
2. Publish a GitHub Security Advisory
3. Credit the reporter (unless anonymity is requested)

We ask that you give us reasonable time to remediate before any public disclosure, and that you avoid accessing, modifying, or exfiltrating data beyond what is necessary to demonstrate the vulnerability.

## Recognition

Responsible disclosures will be acknowledged in the release notes and Security Advisories unless the reporter requests otherwise.
